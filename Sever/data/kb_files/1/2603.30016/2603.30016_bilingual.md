一、引言  
当前主流AI智能体大多基于大语言模型（LLMs）¹构建，易遭受间接提示注入攻击［8］。攻击者可在外部数据（例如检索到的电子邮件、网页内容或第三方工具输出）中嵌入恶意指令，诱导智能体执行有害操作。随着LLM驱动的智能体被部署于风险日益增高的场景，并被赋予更高程度的自主权，提示注入已成为其实现安全、可靠落地的主要障碍。

为应对该威胁，研究界已提出一系列防御方案。模型级防御旨在提升LLM自身的鲁棒性，例如通过面向安全的微调［1］–［3］，或利用其内部表征（如神经元激活模式）检测对抗性输入［10］，［27］。文本级防御则作用于LLM的输入与输出，例如强化提示词设计［9］，或监控文本字符串以识别异常行为［18］。系统级防御［6］，［13］，［14］，［17］，［21］将智能体视为一个完整的计算系统，通过分析语义层面的行为（如工具调用序列、数据访问模式）来检测攻击并实施安全管控。

在上述各类方法中，系统级防御近期受到特别关注，因其可提供更透明的决策过程，并有望给出具有实际意义的安全保障。本文作为一篇立场论文，分享我们对系统级间接提示注入防御的观察与构想。

系统级防御中的另一常见策略是策略强制执行（policy enforcement）[6], [17]：系统预先推理允许的行为，并阻止可疑操作。我们观察到，策略本质上与用户任务及当前规划紧密耦合：重规划通常意味着策略变更。在开放性场景中，事先编写一套完全静态的策略往往困难（甚至不可能）——例如，在调试过程中，无法预先确定哪些文件或日志需要被读取、写入或执行。因此，动态策略更新同样是智能体设计中不可或缺的一环。

然而，我们认识到这种动态设计引发了一个关键安全挑战：外部不可信数据如今可能影响规划与策略。因此，核心研究问题在于：如何区分良性的规划/策略更新与恶意的更新。换言之，我们必须研究如何在重规划与策略更新过程中作出可靠的安全决策。这一问题引出了我们的第二个立场。

**立场 2：大语言模型（LLM）或其他学习型模型对某些安全决策而言是必要的。**  
安全研究人员普遍认为，使用另一个 LLM 作为“裁判”来保障 LLM 智能体的安全并不可靠，因为该裁判模型自身也可能遭受提示注入（prompt injection）攻击而被操控。但与此同时，我们也观察到，智能体执行过程与策略设计的复杂性可能超出传统基于规则的安全策略与机制的表达能力。我们提出通过利用 LLM（或其他学习型模型）的强表达能力来缓解这一张力，但仅限于系统对其可观测内容与可决策范围施加严格限制的场景。也就是说，当必须引入 LLM 参与安全决策时，该模型不应接收任意的（且潜在恶意的）环境文本输入，也不应执行任意任务；而应仅接收经过严格限定范围、结构化的中间产物（structured artifacts），并仅用于受限的特定任务（例如：评估某项提议的规划变更或策略调整是否具有恶意性质）。此类限制可显著削弱攻击者引导模型化决策（进而操控整个智能体）的能力。

此外，这一框架揭示了系统级防御与模型级防御协同设计的重要机遇：借助稳健的系统级设计，我们仅需模型在定义明确、范围狭窄的子任务上具备鲁棒性（例如，能可靠忽略恶意指令），且其输入为结构化数据；而无需其在面对任意不可信输入时完成任意任务。这不仅为模型鲁棒性研究提供了更清晰的方向，也使其成为一个更具可解性的研究课题。

**立场 3：个性化与人类交互应在设计中予以充分考虑，尤其在语义模糊的应用场景中。**  
我们的最终立场强调：部分用户意图、上下文与偏好本质上具有模糊性，无法单靠算法化的系统设计或通用模型改进予以彻底解决。

一类常见情形是语言语义的模糊性。例如，针对用户任务“从我的收件箱中获取并摘要所有紧急邮件”，何谓“紧急”高度依赖于具体用户及其所处情境。此类情形的解决离不开用户的直接参与，也应借鉴可用性安全（usable security）领域的相关洞见。

另一类情形是目标对齐的模糊性。例如，一个编程智能体可能通过不可信内容接收到指令，要求安装某个看似与任务相关、实则恶意或已被攻陷的软件包。此类情况对人类而言亦非罕见：人们常在线查找排障指南并照做，却未加验证。判断此类操作是否可接受，难以仅凭系统级信号进行明确定义与评估，往往需要显式的人类判断或组织层面的策略支持。

上述讨论揭示了算法化、系统级设计的根本局限：开发者必须意识到人类因素在智能体安全中的关键作用。² 除上述三项立场外，我们还指出两个自然衍生的问题：

**问题 1：为何图 1 中缺失关键组件的防御方案在纸面上仍显得有效？**  
我们将此归因于现有基准测试（benchmark）设计的局限性。首先，当前基准缺乏细粒度、多步骤的用户任务，而这类任务恰恰需要重规划（及策略更新）才能成功完成。例如，目前最流行的 AgentDojo 基准 [7] 所含 97 项任务中，仅有 6 项需要策略更新/重规划 [15]。其次，所有现有基准 [7], [15], [26] 均仅考虑非自适应的、静态的攻击载荷（attack payloads）与攻击目标，而非能够针对防御系统动态优化其载荷的自适应攻击者。这种设定容易造成效用性与安全性两方面的虚假乐观，因而亟需构建更真实、更具自适应能力的提示注入基准。

**问题 2：若系统级防御在某些情况下仍需依赖 LLM 和/或人工进行决策，其价值何在？**  
我们认为，系统级防御研究的核心价值在于：它提供了一种结构化的方法，用以分析、理解并增强智能体的安全性。例如，如立场 2 所述，尽管我们在某些安全决策中仍需调用 LLM，但可通过系统设计确保基于 LLM 的裁判器不接收任意环境文本，且仅执行范围受限的判定任务。此外，对 LLM 输入施加结构化约束，也明确了模型级防御的目标：其不再需要抵御任意字符串输入，而可专注于优化针对特定结构化输入与受限判定任务的防御能力。

A. 计划与策略：一种结构化分析与控制大语言模型智能体的方法 从设计上讲，大语言模型（LLM）运行于非结构化的自然语言文本之上。这种非结构化特性有利于大规模LLM训练，却使得基于LLM的智能体难以被形式化分析与精确控制。为应对这一挑战，引入“计划”与“策略”的概念成为一种自然选择。直观而言，**计划描述智能体为完成用户任务所能采取的操作序列，而策略则规定智能体被允许执行的操作范围**。下文给出形式化定义与示例。

**计划（Plan）**：我们将单次智能体执行步骤建模为 $ e := (a, I, O) $，其中 $ a $ 表示具体的智能体动作（例如调用某工具：“读取邮件”），$ I $ 和 $ O $ 分别表示输入与输出对象（例如文件路径，或“最近五封邮件”）。一个计划 $ p \in \mathcal{P} $ 则是若干执行步骤的有序序列（更一般地，可表示为有向图）：$ p := (e_1, \dots, e_T) $，用于刻画完成用户任务的一种可能执行轨迹。

实践中，计划可在不同形式化程度（及不同粒度）上予以表示。例如，针对任务“总结来自Alice的最新一封邮件并草拟回复”，以下是一些示意性表示（非穷举）：

1) 自然语言检查清单：  
　　(i) 检索来自Alice的最新一封邮件

　　(iii) 起草一份礼貌的回复 2) 具有类型化步骤与依赖关系的领域专用语言（DSL）：  
　　e₁: GET_RECENT_EMAIL(sender=Alice, n=1) → emails  
　　e₂: SUMMARIZE(emails) → summary  
　　e₃: DRAFT_REPLY(summary) → draft **策略（Policy）**：策略规定了在执行过程中哪些动作与信息流是被允许的。形式上，可将策略视为作用于执行步骤及其历史的一个谓词 $ \pi(\cdot) $，其中 $ \pi(e_t) = 1 $ 表示在当前上下文中允许执行步骤 $ e_t $。等价地，在计划层面，策略诱导出一组被允许的计划集合 $ \mathcal{P}_\pi \subseteq \mathcal{P} $。

实践中，策略可通过多种类型的约束加以指定。例如，策略可能包括：

- **全局静态访问控制规则**：例如，智能体不得读取、写入或执行用户本身无权访问的数据；  
- **任务特定的最小权限原则**：例如，针对某一特定任务，智能体仅可使用预先指定的一组必要工具，并仅可访问预先指定的一组数据（且不可超出）；  
- **上下文相关的安全原则**：权限可能随当前上下文动态变化；例如，仅当收款方为经验证联系人时，智能体才可发起资金转账；  
- **基于格（lattice-based）的信息流控制（IFC）**：为数据/工具标注安全级别（例如：公开 ⊏ 内部 ⊏ 机密），并仅允许不违反许可流向的动作（例如，禁止从高级别标签向低级别标签写入，即“向下写入”）。

此外，策略亦可跨多个层级进行设定，例如工具层、应用层、网络层、操作系统层及基础设施层。

**B. 系统架构设计** 本小节将逐步介绍我们提出的、面向高实用性与高安全性的智能体构建架构（见图 1），分析其设计空间，并将其与现有防御机制建立关联。

**系统架构概览**：图 1 所示的架构围绕前述“规划”与“策略”概念展开设计。以下为分步说明：

1) 给定一项高层任务，**协调器**（Orchestrator）——通常由大语言模型（LLM）驱动——生成初始的**规划**（即如何完成该任务）与**策略**（即执行过程中被允许的行为边界）；  
2) **规划/策略审批模块**（Plan/Policy Approver）——通常亦由 LLM 驱动——负责监督规划与策略的生成过程。其职责包括：审核所生成的规划与策略，并在检测到不合理成分时向协调器提供反馈。如图中人类图标所示，该模块亦可在必要时将决策权移交至人工用户，以实现个性化配置或澄清模糊目标（详见第 III-C 节第 3 个位置的相关讨论）。从概念上讲，该模块可视为协调器设计的一部分；但我们显式地将其分离出来，以突出其作为**安全决策模块**的核心作用；  
3) **执行器**（Executor）——通常为 LLM——接收规划内容，并生成具体动作（例如：带参数的工具调用）；  
4) **策略执行器**（Policy Enforcer）依据当前策略对拟议动作进行审批或拦截。策略校验既可通过基于规则的程序实现，也可交由 LLM 完成，具体取决于策略的形式化表达方式及其具体实例化方案。对于高风险操作或语义模糊场景，该执行步骤还可进一步交由人工判断（图中以人类图标标示）；  
5) 动作一经批准，即被发送至**运行环境**（Environment）——包括各类 API、网页及文件系统——由其执行并返回响应；否则，策略执行器将向执行器反馈拒绝信息；  
6) 执行器处理来自运行环境或策略执行器的响应，并将结果回传至协调器；此过程可能触发对规划和/或策略的更新；  
7) 智能体基于更新后的规划与策略重复上述循环，直至任务完成。

**系统设计空间**：在该架构框架下，存在两个关键的设计维度。

**设计维度一：策略的作用范围、表示形式与执行机制**。首要设计抉择在于：策略应涵盖哪些约束（例如最小权限原则、全局不变量、上下文相关限制等）；应采用何种形式进行表达（例如模板、类型化领域特定语言 DSL、或显式刻画控制流与数据流的图结构）；以及应如何实施校验（例如程序化检查、基于学习的判定模型、人工监督，或混合机制）。总体而言，策略表达越丰富，其表现力越强，但同时也可能增加校验难度；优秀的设计需在表现力与可验证性之间取得平衡。

**设计维度二：对环境反馈的安全感知式处理（及其引发的规划/策略更新）**。图 1 所示架构允许环境响应随时间推移动态影响规划与策略，从而提升系统适应性；但与此同时，这也扩大了潜在攻击面。因此，第二个关键设计维度在于：在支持此类环境反馈（图 1 中箭头 *a–c* 所示）的同时，严格限制攻击者对模型决策的操控能力。我们在第 III-B 节中进一步探讨具体实现路径。

A. 立场一：面向动态任务与动态环境，必须实施动态（且具备安全意识的）重规划与策略更新 立场一基于一个基本观察：通用型智能体必然运行于动态环境中。因此，为维持高实用性，动态重规划与动态策略更新不可或缺；而如何在保障安全的前提下实现此类更新，则成为一项核心挑战。

在动态环境中，重规划往往对实用性至关重要。一种常见的系统级防御思路是“计划—执行隔离”[6], [23]：即先根据用户任务生成计划，随后基本按原计划执行，避免不可信的环境数据直接干预控制流。尽管该隔离机制可显著降低控制流劫持类攻击的攻击面，但在面对良性环境变化时仍可能表现脆弱。以下提供两个具体示例：

**示例场景（良性的运行时失败）**：某数据分析智能体被要求“从计费 API 中拉取上周销售数据并绘制趋势图”。协调器（orchestrator）计划调用 `GET /v1/invoices?start=...`。运行时，工具返回 `HTTP_410 Gone` 错误，因该端点已被弃用，并由 `/v2/invoices/search` 取代。一个实用的智能体必须能自适应地调整其计划：查阅更新后的文档、切换至新端点，并同步更新数据解析逻辑。若严格遵循计划—执行隔离原则，该智能体将持续重复调用已失效的旧端点，最终无法完成任务。

**示例场景（迭代式调试依赖执行反馈）**：某编程智能体被要求“运行单元测试并修复失败项”。其高层计划（运行测试 → 检查失败日志 → 修改代码）易于表述，但具体下一步操作则高度依赖失败日志所揭示的信息。例如，某一失败可能指向边界检查中的“差一错误”（off-by-one error），而另一失败则可能暴露需重构同步机制的竞态条件（race condition）。在应用补丁后，智能体必须重新运行测试（通常需多次）以确认问题已修复，并检测是否引入新的回归缺陷。由于每次补丁都会改变后续可能出现的失败形态（如有），故难以在任务启动之初就完整预设一份详尽计划；基于执行反馈进行重规划，实为该类任务的内在需求。

**关于条件式/分支式计划的说明**：有人或会主张，在第一个示例中，初始计划可预先嵌入条件与分支逻辑（例如：“若 API 调用成功，则使用返回数据继续执行；否则，查阅更新后的文档”）。此类条件结构虽有助于应对若干预期内的失败模式，但在诸如调试等高度动态的场景中，往往难以事先穷举所有相关运行时错误。此外，当控制流依赖环境所提供的信号时，攻击者可能操纵这些信号以诱导智能体进入特定分支（即“控制流引导”）。因此，孤立的条件式计划既无法充分保障安全性，亦难以满足实用性需求；支持安全感知重规划的系统设计，仍是亟待深入探索的重要研究方向。

除重规划外，动态策略更新同样不可或缺。直观而言，一旦执行计划发生变更，相应策略亦应随之更新。更广泛地看，我们指出：静态策略在实用性与安全性两方面均可能存在明显不足。

**理由一：静态的、任务无关的（如工具级）策略可能过于严苛或过于宽松。**  
我们以 AgentDojo [7] 基准测试中的 `send_money` 工具及 CaMeL [6] 所采用的静态工具级策略为例加以说明。`send_money` 工具用于将用户资金转账至指定收款人；CaMeL 的策略大致规定：仅当同时满足以下两项条件时方可允许调用——（i）收款人信息必须完全源自用户输入；（ii）收款人须被授权查看全部工具参数。该策略在狭窄的基准测试环境中或可奏效，但若作为一项任务无关的全局规则应用于更广泛的现实场景，则易产生误报（false positive）与漏报（false negative）。

**误报示例（过度严苛）**：用户指令为：“请支付 xyz@abc.com 的账单。” 此处收款人信息可能合法地存在于共享媒介中（例如一封电子邮件或一张附带的发票）。然而，该策略僵化地限定收款人信息只能来自用户的初始提示，从而会阻断此类合理的扩展行为。

上述示例表明：静态的、任务无关的策略，有必要在运行时依据实际情境动态优化——或为达成任务目标而适度放宽先前过于悲观的限制（借助确凿证据），或为防控风险而在新威胁浮现时及时收紧先前过于宽松的约束。

在很大程度上保持系统效用：即使无法实现绝对安全，系统也应使低代价攻击难以成功，并显著提高攻击者的实施成本。如此一来，一次成功的入侵将依赖于攻击者具备深厚的专业知识与大量资源投入，而非仅通过注入一条简单的恶意指令即可达成。接下来，我们将阐述为实现这一目标所采取的具体策略。

**核心思想：在大语言模型（LLM）参与安全决策时，避免其直接暴露于原始的、攻击者可控的文本或任意任务中。**  
尽管LLM具有极强的表达能力，但若将其直接用于关键安全决策，则存在显著风险——因其极易遭受提示注入（prompt injection）攻击。因此，我们在系统层面的核心洞见是：设计智能体架构，确保模型**不直接接触**攻击者可任意操控的原始环境文本（例如网页内容、电子邮件或工具输出）。取而代之的是，系统应将不可信的环境反馈转化为**范围明确、结构化程度高**的中间产物——例如带类型的执行轨迹（typed trace）和拟议的计划/策略变更（proposed plan/policy diff），并仅要求模型基于这些受限表示作出判断。换言之，模型应被用作一个**边界清晰的安全决策模块**（例如，在显式约束下评估提议的更新），而非一个面向任意文本、执行任意任务的开放式控制器。这种设计既削弱了攻击者塑造模型上下文的能力，又保留了模型进行语义推理的优势——而这恰恰是严格的程序化检查通常难以实现的。

**协同防御的设计契机：系统级与模型级防御的联合设计。**  
该策略还为系统级与模型级防御的协同设计提供了强有力的机会。系统架构可主动限制并结构化输入至模型的信息，从而为模型级鲁棒性研究指明方向，并使问题更具可解性。在此架构下，所训练的模型不再需要对跨各类任务的任意不可信字符串均具备鲁棒性；相反，它可专注于更小、定义更清晰的问题，例如：判断带类型的变更差异（typed diffs）、针对结构化输出合成校验器（validators）、以及在具备显式失效模式的前提下作出有界决策。我们认为，这种协同设计路径是构建兼具高实用性与高安全性的通用智能体最具前景的研究方向之一。

接下来，我们提出两种具体方案，说明如何在输入与任务受限的前提下，将LLM应用于安全决策。

**方案一：分离“指令识别”与“指令执行决策”。**  
本方案受我们先前关于指令遵循意图分析的工作启发 [12]。我们建议将LLM/智能体的指令遵循过程分解为两个步骤：（1）在上下文中识别候选指令；（2）决定哪些已被识别的指令应当被执行。这种解耦提升了系统透明度，并将部分防御责任从隐式的模型行为转移至显式的系统策略。

**步骤（1）——指令识别：**  
我们建议通过提示工程（prompting）和/或模型训练，使模型**显式地复述**（即重复）其拟执行的指令。其动机在于：经过后训练（post-training）的LLM已在内部执行一种识别过程，用以区分“需遵循的指令”与其他文本；而该识别过程本身才是决定性环节——因为若模型未能将恶意指令识别为“应予执行的指令”，则提示注入攻击即告失败。值得注意的是，我们的目标并非赋予模型一项新能力，而是促使模型将其**已具备的识别能力外显化**，即忠实地复述其本就打算执行的指令。我们假设：训练模型忠实地复述指令，相较于训练其直接识别并忽略所有恶意内容，是一项更易实现且对模型效用损害更小的目标⁴。

**步骤（2）——指令执行决策：**  
我们不在模型层面隐式依赖其鲁棒性，而是在**系统层面**作出指令执行决策。具体而言，系统首先追溯每条被复述指令的来源（例如，通过词/子串匹配，比对可信段与不可信段），随后依据以下任一运行模式执行（作为图1中“策略执行器”（Policy Enforcer）模块的一部分）：

- **模式1（禁止外部指令）：** 若工作流预设指令仅应来自可信源（如用户、开发者或系统提示），则任何被追溯至外部不可信数据的指令默认被拦截。

- **模式2（允许外部指令但需人工审批）：** 若确有必要接受外部指令，系统可在执行每条此类指令（或每一类敏感指令）前，强制要求用户显式确认。

- **模式3（最小化人工干预）：** 若频繁确认不可取，我们建议采用两种互补机制：  
  （a）**预先声明指令预期：** 在处理某一外部信息源之前，智能体先预测该源是否**本应包含可执行指令**（例如，某软件库的“快速入门”页面很可能包含指令）；若预测结果为否，则任何从中提取的指令均按策略拒绝。  
  （b）**结构化指令状态裁决：** 编排器（orchestrator）维护一个结构化的任务/指令数据结构（例如任务树或带类型的执行轨迹），记录已执行指令、当前拟执行指令及其来源。当提出一条新指令节点（例如，“安装库X”）时，系统提交一份结构化的更新请求，交由LLM裁判模块审查。关键在于，该裁判模型仅能查看结构化的当前状态与提议的变更差异（diff），而**完全无法接触原始环境文本**；因此，模型在此被用作一个**有边界的裁决者**（bounded adjudicator），而非对不可信内容进行开放式响应的执行者。

**设计哲学：**  
我们在此强调两项核心设计原则。第一，我们力求将LLM在安全决策中的角色简化至最低限度。只要基于规则的控制手段足以应对，我们就应优先采用——例如，依据来源可信度拦截外部指令（模式1），或强制用户确认（模式2）。

关于现有纯模型级防御机制的说明。模型鲁棒性研究［2］、［3］、［19］已在训练模型以更稳健地执行指令遵循决策方面取得重要进展。然而，从系统安全视角出发，一个关键问题仍未解决：即使模型能够可靠地忽略恶意指令，一旦检测到提示注入（prompt injection），它是否仍应继续依赖同一来源中的其他内容？例如，若模型正确识别出外部文档中存在恶意指令，它是否还应继续使用该文档中其他可能已被攻陷的内容？在多数场景下，答案很可能是否定的。我们认为，这是一个尚未被当前模型鲁棒性研究充分解答的重要安全问题。这也正是我们提出将**指令识别**与**指令执行决策**解耦的原因之一。分离的指令执行决策环节可提升智能体系统的可解释性与安全性，并为已检测攻击的处置方式提供更大灵活性。

提案2：利用大语言模型在运行时合成面向具体步骤的程序化校验器，以验证环境反馈。除口头表述指令及评估计划/指令差异外，系统还可于运行时调用大语言模型，生成确定性的校验逻辑或程序，用于在每一步对环境响应进行验证（即保护图1中的反馈通道a）。在此过程中，模型仅被用于**受控范围内的校验器合成**，而非对原始文本实施任意端到端的任务控制。

示例场景（带字段级完整性约束的网页检索）。假设用户请求：“查找Acme公司的投资者关系页面，并提取第四季度营收数据。” 检索工具返回一个可能含有注入文本的网页。系统可调用大语言模型合成如下校验器：“仅从表头包含‘Quarter’和‘Q4’的表格中，提取‘Revenue’单元格内的营收数值；忽略所有自由段落；所提取数字须符合正则表达式`[0-9,.]+`，并注明对应DOM路径。” 在运行时，系统将该校验器应用于网页的结构化表示（例如DOM转JSON解析结果），自动拒绝违反上述约束的输出，而无需依赖执行器在面对对抗性文本时“自行做出正确判断”。

C. 立场3：在语义模糊的应用场景下，应纳入个性化与人工交互考量 最后，我们的第三项立场强调了纯算法化、系统级防御手段的固有局限：某些安全决策可能取决于语义模糊的语言表达及用户真实意图。在此类情形下，智能体设计必须考虑个性化设置与人工交互机制。下文将讨论两类典型模糊性，同时指出其他具有挑战性且不可避免的模糊形式亦可能存在。

语言语义层面的模糊性。部分任务中包含含义高度依赖用户身份与上下文的术语。此类模糊性即便借助理论上完美的模型也无法消解，因而必须通过人工反馈予以澄清。

示例场景（“紧急”一词具有歧义）。用户请求：“获取并摘要我收件箱中所有紧急邮件。” 此处，“紧急”可能意指“提及24小时内截止期限”，也可能指“被特定标签标记”。该歧义只能通过人工交互加以明确。

目标对齐层面的模糊性。即使某项操作看似与完成用户任务的子目标“相关”，其本身仍可能违背既定安全策略，因而不可接受。

> Question 1: Why do defenses that omit key components
> in Figure 1 still look good on paper? Many existing
> defenses [6], [13], [14], [17], [21] report both high utility
> and strong security in their evaluations, despite omitting key
> design components discussed in this paper. We attribute this
> discrepancy to limitations in existing benchmarks [7], [25]—
> in both user tasks and attack modeling—that can create a false
> sense of security and utility.
> Lack of dynamic, complex tasks and environments that
> require replanning and policy updates. Existing benchmarks
> typically cover only simple tasks with a small number of
> steps (e.g., 3–4) that can be planned ahead (e.g., “read file
> A and put the summary in channel B”). In addition, they
> often fail to model dynamic environments—for example, the
> possibility of encountering runtime errors, as discussed in
> Section III-A. Moreover, the task suite may not be diverse
> enough to expose the limitations of task-agnostic policies
> (recall the send_money example in Section III-A). We
> argue that benchmarks must include longer-horizon, evolving
> tasks and dynamic environments to assess the true utility of
> proposed defenses.
> Lack of dynamic attack objectives. Most benchmark attacks
> focus narrowly on action hijacking to a pre-selected, static
> malicious tool call (e.g., instead of summarizing a webpage,
> the agent is coerced into posting misinformation to Slack). In
> these setups, the attack objective is fixed across user tasks,
> and the attacker-selected tool call often differs substantially
> from the benign one. In practice, however, attacks can be
> more nuanced. For example, an adversary may manipulate
> parameters of the same tool call (e.g., transferring $1000
> instead of $100), or alter the conditions under which an agent
> executes a plan (e.g., “if it is not raining tomorrow, book a
> day hike; otherwise, purchase two movie tickets”). These task-
> and context-dependent objectives are harder to defend against
> and should be reflected in benchmark design.
> Lack of payload optimization. Nearly all benchmark pay-
> loads are static and generated with simple heuristics. Recent
> work suggests that adaptive black-box attacks—for example,
> training a specialized attacker model against a defended sys-
> tem (e.g., via RL) [22], [24], or using iterative optimization
> (e.g., genetic-algorithm-style refinement) [16] to improve the
> payload—can be substantially more effective at surfacing
> prompt-injection vulnerabilities in both models and agentic
> systems. We therefore argue that benchmarks should support
> plugging in adaptive attackers, rather than relying solely on
> fixed attack strings.6
> Question 2: What is the value of system-level defenses if
> they still require LLM and/or human decision making in
> certain cases? System-level defenses are often motivated by
> the promise of security guarantees. Yet Positions 2 and 3 imply
> that some security-critical decisions will still require LLM-
> based judgment and, in inherently ambiguous cases, human
> oversight. This raises a natural question: if we cannot eliminate
> LLMs and humans from the loop, what is the value of system-
> level defenses? We argue that the key value of system-level
> defense research is that it provides a structured and more
> explainable way to analyze, understand, and enhance agent
> security. A well-designed system serves as the skeleton on
> which both LLM- and human-in-the-loop components operate.
> (1) Structured analysis and control via secure interfaces.
> System-level design enables us to replace raw, attacker-
> controlled environment text with narrowly scoped artifacts and
> constrained choices. As discussed in Position 2, the system
> can ensure that the LLM judge does not consume arbitrary
> environment strings directly; instead, the model is invoked
> only for well-defined sub-judgments over structured inputs
> (e.g., a proposed plan/policy diff plus minimal supporting
> evidence). In this way, the system determines what decision
> makers see and what they are allowed to decide, significantly
> reducing the attack surface.
> (2) More targeted and tractable LLM/human research. By
> enforcing structured inputs and bounded judgment tasks, sys-
> tem design makes both model-level and human-in-the-loop
> research more targeted and tractable. Rather than defending
> against arbitrary untrusted strings, model defenses can focus
> on specific structured sub-tasks with clearer success criteria;
> similarly, human-oversight interfaces can focus on decision
> points that are auditable and minimally burdensome. We view
> this kind of system-model-human co-design as a particularly
> promising direction.
> (3) Better explainability, a bridge between rule-based and
> model-based security, and defense in depth. Structured system
> interfaces improve explainability by making it easier to trace
> which evidence, rules, or judgments led to each action. Cru-
> cially, this traceability enables systematic analysis of agent
> failures and identification of weak points. More broadly,
> system-level design serves as a bridge between rule-based
> security enforcement and expressive model-based security-
> decisions, enabling stronger defense in depth without sacri-
> ficing practical utility.

**IV. 讨论**

**问题1：为何图1中缺失关键组件的防御方案在纸面上仍显得效果良好？**  
大量现有防御方案 [6]、[13]、[14]、[17]、[21] 在其评估中均报告了高实用性与强安全性，尽管它们忽略了本文所讨论的关键设计要素。我们将这一矛盾归因于当前基准测试 [7]、[25] 在用户任务建模与攻击建模两方面存在的局限性——这些局限可能营造出一种虚假的安全性与实用性幻觉。

- **缺乏需动态重规划与策略更新的复杂、动态任务与环境。** 当前基准测试通常仅涵盖步骤极少（例如仅3–4步）的简单任务，且这些任务均可预先完整规划（例如，“读取文件A并将摘要发送至频道B”）。此外，它们往往未能建模动态环境——例如，如第III-A节所述，未考虑运行时错误发生的可能性。再者，任务集可能缺乏足够的多样性，无法暴露任务无关型策略（task-agnostic policies）的固有缺陷（参见第III-A节中的 `send_money` 示例）。我们认为，基准测试必须纳入长周期、演进式任务及动态环境，方能真实评估所提防御方案的实用性。

- **缺乏动态攻击目标。** 大多数基准测试中的攻击聚焦于狭义的动作劫持（action hijacking），即强制代理调用某个预设的、静态的恶意工具（例如，本应总结网页内容，却被胁迫向Slack发布虚假信息）。在此类设定下，攻击目标在不同用户任务间保持固定，且攻击者选定的恶意工具调用往往与良性调用存在显著差异。然而在实际场景中，攻击可能更为精细。例如，攻击者可篡改同一工具调用的参数（如将转账金额由$100改为$1000），或改变代理执行计划所依赖的条件（例如，“若明日本地无雨，则预订一日徒步；否则，购买两张电影票”）。这类依赖具体任务与上下文的攻击目标更难防御，理应在基准测试设计中予以体现。

- **缺乏有效载荷优化。** 几乎所有基准测试所采用的有效载荷均为静态生成，且仅依赖简单启发式规则。近期研究表明，自适应黑盒攻击——例如，针对受保护系统训练专用攻击者模型（如通过强化学习RL）[22]、[24]，或采用迭代式优化方法（如基于遗传算法的精细化调整）[16] 来提升有效载荷质量——在揭示大语言模型及具身智能体系统中的提示注入漏洞方面，显著优于静态攻击手段。因此，我们主张基准测试应支持接入自适应攻击者，而非仅依赖固定攻击字符串。⁶

**问题2：若系统级防御在某些情形下仍需依赖大语言模型（LLM）和/或人工决策，其价值何在？**  
系统级防御常以提供安全保障为出发点。然而，立场2与立场3表明：部分安全关键型决策仍将依赖LLM判断；而在本质上存在模糊性的场景中，则需人工监督。这自然引发一个疑问：倘若我们无法将LLM与人类完全排除出决策闭环，那么系统级防御的价值究竟何在？我们认为，系统级防御研究的核心价值在于：它为分析、理解与增强智能体安全性提供了结构化且更具可解释性的路径。一个设计良好的系统，构成了LLM与人在环（human-in-the-loop）组件协同运作的骨架基础。

（1）**通过安全接口实现结构化分析与控制。** 系统级设计使我们得以将原始的、由攻击者可控的环境文本，替换为范围明确的中间产物与受限的选择空间。如立场2所述，系统可确保LLM判别器不直接处理任意环境字符串；相反，该模型仅被调用于面向结构化输入的明确定义子判别任务（例如，对一项拟议计划/策略变更及其最小必要支撑证据进行评估）。由此，系统决定了决策者所见内容及其被允许作出的判断，从而大幅缩减攻击面。

（2）**更聚焦、更可行的LLM/人工研究路径。** 通过强制采用结构化输入与边界清晰的判别任务，系统设计使得模型层面与人在环层面的研究均更具针对性与可行性。模型防御不再需要应对任意不可信字符串，而可集中于具有明确定义成功标准的具体结构化子任务；同理，人工监督界面亦可聚焦于可审计、负担最小化的关键决策节点。我们将此类系统—模型—人类协同设计视为极具前景的研究方向。

（3）**更强的可解释性、规则驱动与模型驱动安全之间的桥梁，以及纵深防御能力。** 结构化的系统接口提升了可解释性，使其更易追溯每一项动作背后的证据依据、规则应用或判断过程。尤为关键的是，这种可追溯性支持对智能体失效案例开展系统性归因分析，并精准识别薄弱环节。更广泛而言，系统级设计架起了规则驱动型安全执行与表达力更强的模型驱动型安全决策之间的桥梁，在不牺牲实用性的前提下，实现更稳健的纵深防御。

**V. 核心启示与结论**  
本文提出了我们关于系统级防御的总体构想。我们阐明了三项核心立场：其一，亟需支持动态重规划与策略更新；其二，亟需引入基于模型的安全决策机制；其三，亟需将人类因素纳入整体安全架构考量。

3. 当必须使用 LLM 做出安全决策时，我们应严格约束其输入内容及其决策范围（即：模型可观测的信息范围，以及被允许做出决定的范畴）。此举可显著缩小攻击面，并推动更具针对性的模型级鲁棒性研究——聚焦于受限输入与受限任务下的鲁棒性，而非面向攻击者可控环境文本所发起的任意任务的鲁棒性。
4. 人类监督对于代理的可靠使用仍属不可或缺，尤其因为模糊情形（ambiguous cases）不可避免。其中一项核心挑战在于：在不牺牲安全性或实用性前提下，尽可能降低人工干预的负担。

除上述核心主张外，我们还探讨了现有代理安全基准测试（agent-security benchmarks）的局限性，并厘清了系统级防御（system-level defenses）的作用。我们认为，系统级设计通过整合程序化安全规则、基于模型的安全决策以及人工干预，为保障 LLM 代理安全提供了骨架与支柱。

我们希望本文能推动关于“系统–模型–人类协同设计”（system–model–human co-design）的进一步研究，从而构建兼具高安全性与高实用性的智能代理。
